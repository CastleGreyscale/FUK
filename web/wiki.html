Overview

FUK (Framework for Unified Kreation) is a local-first AI rendering pipeline built for VFX professionals. It provides a structured, stable environment for image and video generation that integrates cleanly into existing production pipelines — without telemetry, cloud dependencies, or workflows designed around consumer use cases.

FUK runs entirely on your hardware. Models, outputs, and project data stay on your machine.

Design Philosophy

Most AI generation tools are built around exploration — infinite scroll, quick iterations, no project structure. FUK is built around production. Every output lives in a project/shot/version hierarchy. Every generation captures complete metadata. Nothing is ephemeral unless you want it to be.

The pipeline is intentionally minimal in its interface and maximal in its output. Controls are exposed where they matter. Defaults are sane. The goal is repeatability and composability with downstream tools, not hiding complexity behind a friendly UI.

What FUK Is Not

  • Not a cloud service — no API keys, no usage limits, no data leaving your machine
  • Not a consumer tool — designed for pipeline integration, not social sharing
  • Not a model trainer — FUK generates; training happens in dedicated tools
  • Not locked to one model family — new pipelines can be added via configuration

FUK is in active development. Core generation is stable. AOV outputs are functional but being refined. See the roadmap section on the main page for current focus areas.

Installation

Prerequisites

  • Python 3.10+
  • CUDA 11.8+ (NVIDIA) or compatible AMD drivers
  • Git
  • FFmpeg

Steps

  1. Clone the repository:
    git clone https://github.com/CastleGreyscale/FUK
  2. Install dependencies:
    pip install -e .
  3. Run setup:
    chmod +x setup.sh && ./setup.sh
  4. Configure models — edit config/models.json to select which models to use, remove entries for any you don't want
  5. Set model root — edit config/defaults.json and set the model root directory to wherever your models live
  6. Download models:
    ./download_models.sh
  7. Launch FUK:
    ./start.sh

First Launch

On first launch FUK will verify model paths and report any missing files. The interface will open at http://localhost:7860 by default. If a model fails to load it will appear grayed out in the model selector — check the terminal output for the specific error.

See the GitHub README for the complete and up-to-date installation instructions, including Windows/WSL2 setup.

VRAM Considerations

VRAM requirements scale with model size and output resolution. As a rough guide:

  • 8GB — base image generation at moderate resolutions; some models will not load
  • 12GB — comfortable for most image generation pipelines
  • 16GB+ — video generation, high-resolution image gen, AOV passes
  • 24GB — full pipeline with all features; recommended for production use

FUK will attempt to offload to system RAM when VRAM is insufficient, but generation will be significantly slower. The VRAM Management preset in each generation panel controls the offload strategy.

Disk Space

  • Base install: ~15GB
  • Models: 50GB–500GB+ depending on selection — see File Locations for where models are expected
  • Outputs: variable; EXR files with all AOV passes are substantially larger than standard image formats

Model and application install directories can be on separate drives. See the Windows Storage Doc for model location considerations on Windows.

Detail Bias

The sampler runs a denoising schedule from high noise (sigma ≈ 1.0, where global composition is decided) down to clean (sigma ≈ 0, where fine texture and edges form). At Detail Bias 1.0 the full schedule runs. Below 1.0, the high-noise front is truncated and the same step count is repacked into a smaller, lower-noise window — so a larger share of compute lands in the texture-forming band.

The result is grittier surfaces, more visible micro-detail, and slightly less of the "diffusion smoothness" sheen. Composition becomes more conservative (the model has less room to wander globally), which makes this knob pair naturally with reference images and ControlNet conditioning where you didn't want global drift anyway.

The slider is clamped to 0.85–1.00; values below 0.85 strip too much of the composition phase and produce broken output. Try 0.85–0.90 for noticeable grit, 0.95 for a subtle bump.

Image Input & ControlNet

Edit and context models accept one or more reference images as conditioning input. Load a reference image via the Media Input component, select a preprocessor to extract a conditioning signal, and set the edit strength. See Preprocessors for available preprocessing modes.

Batch Generation

FUK supports batch generation — queue multiple runs from the same settings in a single pass. Each result is saved as its own versioned output with its own metadata sidecar. Batch jobs run sequentially and can be monitored from the generation modal.

Output Format

Video is saved as a PNG frame sequence plus a composited MP4 for preview. The frame sequence is the primary output — import it directly into Nuke, After Effects, DaVinci Resolve, or any compositing application. EXR passes are generated per-frame when AOV export is enabled via the Layers Panel.

Frame rate can be increased post-generation via FILM interpolation. See Frame Interpolation. Interpolation runs on the frame sequence, not the MP4.

Preprocessors

Preprocessors convert input images into conditioning signals that guide generation — giving precise control over pose, depth, structure, and edges in the output without fighting the model through prompt engineering alone. Preprocessing is done in the Preprocess Panel; the output is then used as a ControlNet conditioning input in the Image Panel.

Canny Edge Detection

Extracts hard edges and structural lines from an input image. Useful for preserving architectural forms, hard-surface geometry, or any scene with defined edges that should carry through to the output.

ParameterDescription
Low ThresholdMinimum gradient magnitude to consider as an edge — lower values detect more edges
High ThresholdMinimum gradient magnitude for a strong edge — edges below this but above low threshold are kept only if connected to a strong edge
Blur KernelPre-detection smoothing — reduces noise at the cost of fine edge detail
InvertFlips the edge map — white edges on black vs. black edges on white

OpenPose

Detects human body pose from an input image and extracts a skeleton map. Use this to replicate or constrain body positions across generations. Useful for character work where posture and gesture need to be consistent shot to shot.

ParameterDescription
BodyFull body keypoint detection — always enabled
HandsAdds wrist, palm, and finger keypoints
FaceAdds facial landmark keypoints

Depth Estimation

Generates a depth map from an input image using Depth Anything v3, encoding near-to-far distance as a greyscale gradient. Used for spatial conditioning — maintaining scene depth structure across generations.

ParameterDescription
ModelDA3 model variant — larger models produce more accurate depth at higher cost: da3_mono_large, da3_metric_large, da3_giant
InvertFlips near/far encoding
NormalizeStretches depth values to fill the full 0–1 range
Range Min / MaxClamps the depth range — useful for isolating a specific depth band

Workflow

  1. Go to the Preprocess Panel and load a reference image or video
  2. Select a preprocessing method and configure its parameters
  3. Generate — the conditioned output is saved alongside your source
  4. In the Image Panel, load the preprocessed result as a ControlNet input
  5. Set conditioning weight and guidance range, then generate

Preprocessors also run on video sequences — the output is a per-frame conditioning sequence. Depth + pose is a common combination for character shots.

Regional Conditioning (EliGen)

EliGen integration brings regional conditioning directly into the generation pipeline. Define compositional zones within the frame and assign independent parameters to each — separate prompts, weights, and conditioning per region. This gives you direct spatial control over composition without relying entirely on prompt engineering.

How It Works

Rather than sending a single prompt to the model and hoping for a specific composition, EliGen lets you explicitly partition the frame. Each zone is a mask that defines a spatial area, paired with a prompt and weight that apply only within that area. Zones are composited during the denoising process — the model integrates them into a coherent result rather than pasting regions together.

EliGen is activated by selecting a Qwen Image + EliGen model in the Image Panel. Zone masks are loaded as .psd or .ora files — each layer in the file becomes a separate conditioning zone.

Zone Parameters

ParameterDescription
Entity MasksPath to a .psd or .ora file — each layer is treated as a separate zone mask
Context ImageOptional reference image for the overall scene; set EliGen alpha (0.0–2.0) to control its influence
EliGen AlphaStrength of the context image's influence on the full generation (0.0 = no influence, 2.0 = strong)
Mask StrengthPer-zone conditioning strength

Practical Notes

  • The global prompt still applies to the whole frame — zone prompts add to it rather than replace it
  • Zones work best for broad compositional control (foreground subject, background environment, sky) rather than fine detail placement
  • EliGen and ControlNet can be used together — ControlNet handles structural conditioning, EliGen handles content per region
  • Create zone masks in any application that exports layered PSD or ORA files — Photoshop, Krita, and Affinity Photo all work

Upscaling

FUK integrates Real-ESRGAN for post-generation upscaling. Apply to single images or to video frame sequences. Upscaling runs as a separate pass in the Postprocess Panel — it does not require re-running the full generation pipeline, and the original output is always preserved.

Available Models

Real-ESRGAN x2 PlusModerate upscale with less processing time2×
Real-ESRGAN x4 AnimeIllustrated, stylized, or anime-adjacent output4×
RealESR-General-x4v3General-purpose; handles a wider variety of content types4×

Workflow

  1. Go to the Postprocess Panel and select the Upscale tab
  2. Load an image or frame sequence
  3. Select an upscaling model and scale factor
  4. Run — the upscaled result is saved as a new version alongside the original

Upscaling Video

When applied to a video sequence, upscaling runs frame by frame. Processing time scales linearly with frame count. The output is a new upscaled frame sequence saved in the same version directory.

Frame Interpolation

FILM (Frame Interpolation for Large Motion) synthesizes intermediate frames to increase the effective frame rate of generated video. Useful for smoothing motion, converting a 12fps generation to a 24fps deliverable, or creating slow-motion sequences from standard-rate output. Interpolation runs in the Postprocess Panel.

How FILM Works

FILM uses a learned model to estimate motion between adjacent frames and synthesize intermediate frames that are temporally consistent. Unlike traditional optical flow methods, it handles large displacements and occlusions more gracefully — making it a better fit for AI-generated content where frame-to-frame motion can be irregular.

Parameters

ParameterDescription
Source FramerateThe frame rate of the input sequence — must match the actual rate the sequence was generated at
Target FramerateDesired output frame rate — FILM synthesizes however many intermediate frames are needed to reach it
Video Output ModeWhether to produce an MP4 preview alongside the output frame sequence

Practical Notes

  • FILM performs best with smooth, continuous motion — highly chaotic or flickering content will not interpolate cleanly
  • Generated at 12fps → interpolated to 24fps is the most common workflow
  • Interpolation does not generate new content — it synthesizes in-between frames; it cannot recover motion that wasn't present in the original generation
  • Interpolation runs on the frame sequence, not the MP4 preview — load the sequence directory, not the video file

AOVs & Passes

FUK generates Arbitrary Output Variables (AOVs) alongside the beauty render via the Layers Panel. These technical passes enable compositing, grading, and integration with professional DCC tools downstream.

AOV outputs are currently being refined. Quality and coverage are improving across builds. Feedback on specific issues is appreciated — contributions welcome.

Available Passes

PassDescriptionChannelsStatus
BeautyMain generation output — full color renderRGBStable
AlphaSubject mask / transparency channelAStable
DepthScene depth encoded as normalized 32-bit float — closer subjects brighterZIn refinement
NormalsSurface normal directions in world or tangent space — RGB encodes XYZ orientationRGBIn refinement
CryptomatteObject and material segmentation masks compatible with the Cryptomatte standard, generated via SAM2RGBA × layersIn refinement

Using AOVs in Compositing

AOV passes are exported together in a single multi-layer EXR file — see EXR Export for the layer structure. Typical uses:

  • Depth — depth-of-field in comp, Z-based fog, depth sorting of elements
  • Normals — relighting in comp, surface-aware grading, integration with CG elements
  • Cryptomatte — instant object isolation, per-element color correction, matte extraction without manual rotoscoping
  • Alpha — subject extraction, compositing over new backgrounds

EXR Export

FUK exports multi-layer EXR files containing all available AOV passes in a single file via the Export Panel. EXR is the standard interchange format in professional VFX and compositing pipelines — it preserves full floating-point precision and carries all passes in one file.

Layer Structure

Layer NameChannelsDescription
RGBAR, G, B, ABeauty render with alpha
depthZDepth pass
normalR, G, BSurface normals
crypto_objectR, G, B, ACryptomatte object layer
crypto_materialR, G, B, ACryptomatte material layer

Export Options

  • Multi-Layer EXR — all passes combined into a single file per frame
  • Single-Layer EXRs — one file per pass per frame; useful when downstream tools don't support multi-layer EXR
  • Bit Depth — 8, 16, or 32-bit per channel; 32-bit recommended for technical passes
  • Compression — ZIP, PIZ, RLE, and others; PIZ is efficient for noisy data, ZIP for smooth gradients

Compatible Applications

  • Nuke (full Cryptomatte support via the CryptomatteNode)
  • Fusion / DaVinci Resolve (Cryptomatte operator)
  • After Effects (with ProEXR for multi-layer access)
  • Blender Compositor (native multi-layer EXR and Cryptomatte)
  • Houdini COPs
  • Katana

LoRA Dataset Builder

The LoRA Dataset Builder generates training datasets for custom LoRA models using FUK's own generation pipeline. Rather than sourcing and curating images manually, it generates a structured set of variations from a subject description — covering pose, lighting, and context diversity needed for effective LoRA training. The tool lives in the Utilities tab.

Workflow

The builder runs in three phases:

  1. Setup — configure the subject, model, and generation parameters
  2. Generation — FUK generates the full variation set; progress is tracked in real time with a cancel option
  3. Curation — review generated images and vote thumbs up / thumbs down on each; finalize saves only the approved images

Setup Parameters

ParameterDescription
Subject TypeCategory preset: character, object, or environment — sets the variation pack and captioning approach
Subject NameName or token for the subject; used as the trigger word in generated captions
Preset PackVariation pack for the selected subject type — determines what poses, lighting, and contexts are generated
Variation DeselectionUncheck specific variations from the pack to skip them
Seed StrategyFixed — all variations use a single seed for consistency; Random — each variation gets its own seed
LoRAOptional: generate using an existing LoRA to produce in-style dataset images
LoRA AlphaBlend weight for the LoRA if one is selected
Steps / CFG ScaleGeneration quality parameters — same as the Image Panel

Curation Phase

After generation completes, every image is presented for review. Thumbs up keeps the image in the final dataset; thumbs down excludes it. Finalize writes only the approved images and their captions to the output directory. Rejected images are not deleted — they remain in a rejected/ subdirectory in case you want to revisit them.

Output Structure

lora_datasets/[subject_name]/
├── approved/
│   ├── 0001.png
│   ├── 0001.txt       ← caption file, one per image
│   ├── 0002.png
│   └── ...
├── rejected/
│   └── ...
└── dataset.json       ← subject name, preset, seed strategy, generation log

Captioning

Captions are generated automatically for each image based on the subject name and variation context. The caption format is compatible with most LoRA training pipelines (kohya_ss, SimpleTuner, etc.). Captions can be edited manually after generation — open the caption editor from the dataset panel before finalizing.

App Layout

The FUK interface is a single-page application organized around a tab bar at the top, a project bar below it, and a collapsible generation history panel on the right. Each tab corresponds to a stage in the pipeline — from generation through post-processing and export.

Tab Overview

VideoImage-to-video and controlled video generation

Typical Pipeline Flow

  1. Image or Video tab — generate the primary output
  2. Preprocess tab (optional) — extract conditioning maps if you need ControlNet on the next generation pass
  3. Layers tab (optional) — generate AOV passes from the output
  4. Postprocess tab (optional) — upscale or interpolate
  5. Export tab — assemble all passes into a deliverable EXR

Result Passing Between Tabs

Results from any tab can be dragged from the Generation History panel and dropped onto the input area of any other tab. The output path is inserted automatically. This is the primary way to chain pipeline steps — no manual file picking required.

Playback Speed

A global playback speed control (6–48 FPS) is available in the main app header. It applies to video previews across all tabs — set it to match the frame rate of your generated sequences for accurate preview timing.

Project Bar

The Project Bar sits below the tab bar and persists across all tabs. It manages the three-level hierarchy — project / shot / version — that all FUK outputs are organized under. Every save operation writes to the currently selected project, shot, and version.

Hierarchy

  • Project — top-level container; typically a film, commercial, or personal project. Maps to a directory under outputs/
  • Shot — a discrete sequence or scene within the project. Maps to a subdirectory under the project
  • Version — an individual saved result within a shot. Versions increment automatically; each version gets its own directory containing the output files and meta.json sidecar

Controls

Shot dropdownSwitch between shots in the current project
Version dropdownSwitch between saved versions of the current shot; selecting a version loads its metadata back into the active tab
SaveSave the current generation result to the active project/shot/version
Version UpIncrement the version number and save — preserves the previous version
New ShotCreate a new shot within the current project
New ProjectCreate a new project directory

Unsaved Changes

When a generation result is present but not yet saved, the Project Bar displays an unsaved changes indicator. Switching shots or versions without saving will prompt for confirmation.

Media Input

The Media Input component (MediaUploader) is the standard file input throughout FUK. It appears in every tab that accepts an image, video, or sequence as input. All instances share the same behavior and supported formats.

Input Methods

  • Drag and drop — drag files or folders directly from your file manager onto the drop zone
  • File dialog — click the upload area to open a native file picker (via the backend tkinter dialog); supports multi-select
  • History drag — drag an item from the Generation History panel directly onto a Media Input to load that output's path without manual file picking
  • Path paste — paste a file path directly into the input field

Dimension Detection

When an image or sequence is loaded, FUK reads its dimensions automatically. Tabs that derive their output resolution from the input (Video, Preprocess, Postprocess) update their dimension display immediately. The Image tab displays a warning if the input dimensions don't align to the model's required 16px grid, and offers a "Pad to [size]" button to adjust automatically.

Seed Control

The Seed Control component appears in the Image and Video tabs. It manages the randomness seed used during generation — the single most important parameter for reproducibility.

Modes

ModeBehavior
RandomA new random seed is generated each time you generate. Use this when exploring — every result will be different even with identical settings
FixedThe same seed is used for every generation. Combined with identical parameters, this produces identical results — the basis for repeatable output
IncrementThe seed increments by 1 after each generation. Useful for batch exploration — you get a sequence of related but distinct results

Controls

  • Seed input — visible in Fixed and Increment modes; enter any integer
  • Randomize button (dice icon) — generates a new random seed and puts it in the field; available in Fixed and Increment modes so you can start from a random point while still locking in repeatability
  • Save to favorites (star button) — saves the current seed with an optional label; saved seeds persist across sessions
  • Saved seeds dropdown — load any previously saved seed in one click

Favorites

Favorite seeds are stored in local settings and persist across sessions. Add a descriptive note when saving a seed — the note appears in the dropdown, making it easy to identify which seed produced a specific result without having to cross-reference the History Panel.

Image Panel

The Image tab is the primary generation interface. Its layout is divided into a preview area on the left and a column of parameter cards on the right.

Preview Area

Displays the current generation result in a zoomable, pannable viewer. Shows the output dimensions and generation time. When no result is present, the placeholder displays the configured target dimensions. History items can be dragged from the Generation History panel and dropped here to reload their metadata into all parameter cards.

Image Tools Card

Visible when an edit or context model is selected. Contains controls for reference image input:

  • Control Images — Media Input for one or more reference images used as conditioning; supports multi-image upload
  • Edit Strength — how much the model deviates from the reference (0.0 = identical to reference, 1.0 = freely interprets); range 0.0–1.0
  • Context Image (EliGen models) — optional scene-level reference; EliGen alpha (0.0–2.0) controls its influence
  • Entity Masks (EliGen models) — path to a .psd or .ora file containing per-zone masks; each layer becomes an independent conditioning zone

Model & Style Card

  • Model dropdown — selects the active image model; the available list is populated from config/models.json; settings are persisted per-model so switching models doesn't lose your configuration for either
  • LoRA section — add multiple LoRAs via the add button; each LoRA has its own dropdown (selects from models/loras/), multiplier (0.0–2.0), bypass toggle, and remove button; bypass lets you quickly A/B a LoRA without removing it from the list

Generation Card

  • Steps — preset buttons (20, 40) or custom input (1–100)
  • Guidance Scale — prompt adherence, 1.0–10.0
  • Exponential Shift Mu — optional; adjusts noise schedule shift; leave blank for model default or enable auto-calculation
  • VRAM Management — preset dropdown controlling the model offloading strategy; higher offload settings reduce VRAM usage at the cost of generation speed

Dimensions Card

For text-to-image models:

  • Aspect Ratio — dropdown of common ratios; changing it recalculates height automatically
  • Width — input in pixels (512–2048), snaps to 16px; height is calculated from the aspect ratio

For edit/context models, the Dimensions card instead shows the source image dimensions and flags any 16px alignment issues with a "Pad to [size]" button.

Prompt Card

Auto-resizing textareas for the positive prompt and negative prompt. Both expand as you type — no fixed height limit.

Footer

Generate button (disabled until a prompt is entered), batch count control, and a generation progress indicator. Clicking Generate opens the generation modal with a live console and progress bar.

Video Panel

The Video tab drives Wan I2V and Fun Control video generation. Its layout mirrors the Image tab — preview on the left, parameter cards on the right.

Preview Area

Preprocess Panel

The Preprocess tab converts input images or videos into conditioning maps — depth, pose skeleton, or edge maps — for use as ControlNet inputs in the Image tab. Preprocessing is a separate step from generation so you can reuse a conditioning map across many generations without recomputing it.

Layout

A method selector at the top (Canny / OpenPose / Depth) switches between preprocessing modes. Below it: an input section with a Media Input, method-specific parameter controls, and a before/after result preview.

Video Mode

When a video or frame sequence is loaded, the panel switches to video mode. Preprocessing runs per-frame and outputs a new sequence. The output mode setting controls whether the result is saved as an MP4, a frame sequence, or both.

Method Parameters

See Preprocessors for full descriptions of each method's parameters.

Result Display

The before/after preview shows the original input alongside the preprocessed output. For video, a frame counter lets you step through the sequence to verify quality before committing to a full run.

Postprocess Panel

The Postprocess tab handles upscaling and frame interpolation. Both operations accept the same input types (image, video, or sequence) and save their output as a new versioned result alongside the original.

Upscale Tab

  • Input — Media Input; accepts images, videos, or frame sequences
  • Method — upscaling model selector (Real-ESRGAN variants)
  • Factor — scale multiplier: 1×, 2×, 3×, or 4×

The result preview shows a before/after comparison. For video, the comparison is shown at the current frame. See Upscaling for available models.

Frame Interpolation Tab

  • Input — frame sequence or video file
  • Method — interpolation model (RIFE)
  • Source Framerate — the frame rate of the input sequence; must be set correctly for the output to play at the right speed
  • Target Framerate — desired output frame rate; FILM synthesizes as many intermediate frames as needed
  • Video Output Mode — whether to generate an MP4 preview alongside the output sequence

See Frame Interpolation for more detail.

Layers Panel

The Layers tab generates AOV passes — depth maps, surface normals, and Cryptomatte segmentation — from an existing image or video. This is a post-generation step: you bring your finished beauty output here and extract the technical passes needed for compositing.

Input

A Media Input at the top of the panel accepts a single image, video, or frame sequence. Drop a result from the Generation History to load it automatically.

Layer Selection Card

Checkboxes for each available pass: Depth, Normals, Cryptomatte. Only the checked passes are computed — disable passes you don't need to save processing time.

Depth Settings Card

  • Model — Depth Anything v3 variant: da3_mono_large, da3_metric_large, da3_large, da3_giant; larger models are more accurate but slower
  • Invert — flips near/far encoding
  • Normalize — stretches depth values to fill the 0–1 range
  • Range Min / Max — clamps the output depth range; useful for isolating a specific depth band

Normals Settings Card

  • Method — from_depth derives normals from the depth pass; scratch estimates normals directly from the image
  • Depth Model — which DA3 variant to use for the depth step (only relevant when method is from_depth)
  • Space — tangent (surface-relative) or world (camera-relative) normal encoding
  • Flip Y — flips the Y axis of the normal map; required by some compositing applications
  • Intensity — normal map contrast/strength (0.0–2.0)
  • Num Iterations — refinement passes for the normal estimation (1–20); more iterations improve accuracy at the cost of speed
  • FOV Degrees — assumed camera field of view used in the depth-to-normal conversion (1–180)
  • First Frame Only — when processing video, compute normals only for the first frame; useful for quick checks before a full run

Cryptomatte Settings Card

  • Model — SAM2 variant used for object segmentation (sam2_hiera_large)
  • Max Objects — maximum number of distinct objects to segment and track
  • Min Area — minimum object size in pixels; objects smaller than this threshold are ignored

Result Display

Generated passes are shown as individual previews below the settings. Each pass can be inspected at full resolution before being passed to the Export Panel.

Export Panel

The Export tab assembles beauty and AOV pass images into multi-layer EXR files ready for professional compositing pipelines. Each pass is loaded independently — you can mix outputs from different generation runs if needed.

Layer Inputs Card

Four inline image inputs, one per pass:

  • Beauty — the primary generation output (RGBA)
  • Depth — depth pass from the Layers Panel
  • Normals — normals pass from the Layers Panel
  • Crypto — Cryptomatte pass from the Layers Panel

Only layers with a loaded input are included in the export. Drop passes from the Generation History to load paths automatically. When sequence inputs are detected, the panel switches to video mode and exports per-frame EXRs.

Save Location Card

  • Export path — destination directory; click Browse Folder to select via file dialog
  • Create New Folder — create a subdirectory within the export path in one step
  • Filename pattern — output filename template; supports {name} (source filename without extension) and {frame:04d} (zero-padded frame number)
  • Sequence start frame — the frame number assigned to the first file in the export; useful when offsetting into a larger sequence

Export Format Card

  • Multi-Layer EXR — all passes in a single file per frame; the preferred format for professional compositing
  • Single-Layer EXRs — one file per pass per frame; use when your compositing application doesn't support multi-layer EXR

EXR Settings Card

  • Bit Depth — 8, 16, or 32-bit per channel; 32-bit recommended for depth and normals passes to preserve floating-point precision
  • Compression — ZIP (good for smooth gradients), PIZ (efficient for noisy/grainy content), RLE (fast, lower compression ratio)
  • Color Space — Linear, sRGB, or ACEScg; use Linear for technical passes, match your project's color management for the beauty

Channels Card

A visual guide showing which input layer maps to which EXR channel group — useful for verifying the layer mapping before committing to a full export run.

Spec Tool

The Spec Tool lives in the Utilities tab. AI generation spec dimensions have non-obvious constraints — models are trained on specific aspect ratios and resolution buckets, and going outside those ranges degrades output quality. The Spec Tool provides a guided way to create valid generation specs and downloadable composition guide templates.

Spec Configuration

  • Aspect Ratio — preset dropdown (1:1, 4:3, 16:9, 1.85:1, 2.35:1, 2.39:1, 21:9) or custom W:H input
  • Resolution Width — input in pixels; height is calculated from the aspect ratio; snaps to 16px alignment automatically
  • Frame Count — for video specs; snaps to the 4n+1 pattern required by Wan video pipelines

Composition Guides

Overlay composition guides on the canvas preview and burn them into the downloadable template:

  • Rule of Thirds
  • Golden Spiral
  • Vanishing Points
  • Action Safe
  • Title Safe
  • Center Cross

Guide color and background color are configurable via color pickers. Background alpha controls transparency of the canvas (set to 0 for a transparent PNG template).

Actions

  • Download PNG — exports the canvas at the configured resolution with the selected guides burned in; use as a layout reference or import into your generation workflow as a composition guide image
  • Save Blender Template — exports the spec as a Blender scene template at the target resolution and frame count

Burn In

When enabled, frame numbers are rendered onto each frame of the Blender template output — useful for editorial reference when working with a delivery spec that has a specific frame count.

Prompt Panel

The Prompt Panel is the floating control in every Image and Video tab where you author the positive and negative prompt. It is more than a textarea: it unifies your saved prompt tags, the trigger words of installed LoRAs, and caption analyses exported from the LoRA builder into a single shorthand vocabulary. From that vocabulary you have three ways to build a prompt — fast inline shorthand, structured slot picks, or an LLM refinement — and they all share the same source of truth.

The Three Modes

  • Inline shorthand (# autocomplete) — type # in the prompt textarea and a dropdown of matching tokens appears above the field. Selecting one inserts a marker (e.g. #noir) at the caret. Markers stay in the text as you write — they're expanded later on Compile or Expand.
  • Slot picker — toggle Slots in the header to open a category grid of available tokens. Drilling into a category lists every token in it; clicking one drops it into the Style chip tray below the prompt rather than into the textarea inline. This keeps subject/action language in the main field and accumulates style intent separately so Compile can produce a disciplined two-paragraph prompt.
  • LLM refine (Expand) — sends your current draft plus style chips plus the same vocabulary to qwen3-vl:8b and asks it to refine the text into a cinematographer-grade prompt. The model is instructed to use vocabulary phrases verbatim and preserve your existing intent rather than rewrite from scratch.

The Vocabulary

Every dropdown and slot list draws from one combined token list. Source badges identify where each came from:

  • tag — your saved entries from the Describe & Tag tool. Edit them at fuk/ui/data/prompt_tags.json or via the Describe & Tag UI.
  • lora — synthesized from fuk/config/defaults_loras.json. Each entry's trigger_word becomes a marker, and its inject_text becomes the expansion. The list is filtered by the current model's family (a Qwen tab won't see Wan-only LoRAs).
  • phrase — top bigrams and trigrams imported from caption analyses dropped in fuk/ui/data/lora_captions/ by the standalone LoRA builder. These only appear when the matching LoRA is active in the model panel. They have no marker — they insert as raw text since they're already short natural-language phrases.

Compile (Deterministic)

Compile expands every #marker in the textarea against the vocabulary, then appends a trailing Style: … sentence built from the chip tray. The result is written back to the textarea and the chip tray clears. No LLM is involved.

  • If the textarea already ends in . ! or ? the punctuation isn't doubled.
  • A blank line separates the main paragraph from the Style: sentence so the textarea reads as two paragraphs.
  • Unknown markers (no matching tag or LoRA) stay in place and surface as a warning under the textarea.
  • If an active LoRA's trigger word never appears in the prompt, the response includes missing_lora_triggers — the UI doesn't auto-inject anything, but it's there so a future check could surface it.

Expand (LLM)

Expand runs the same Compile step internally to produce a concrete draft, then sends that draft along with the vocabulary list to qwen3-vl:8b. The system prompt instructs the model to:

  • Refine the draft rather than rewrite it
  • Use vocabulary phrases verbatim where they apply
  • Preserve the Style: sentence as authoritative
  • Add motion language for Video tabs; suppress it for Image tabs
  • Return JSON: { positive, negative, notes? }

On success the textarea is replaced with the model's positive prompt, the negative prompt is updated only if the model provided one, the chip tray clears, and any notes appear as a dismissible note under the warning row. Requires Ollama running and qwen3-vl:8b pulled — see Describe & Tag for setup. Every request uses keep_alive: 0 so the LLM is evicted from VRAM immediately and the generation pipeline gets it back.

Style Chips

The chip tray appears below the prompt textarea whenever slot picks accumulate. Each chip shows the token's display name; hover for its expansion. Click the × on a chip to remove it, or clear on the tray header to drop them all. Chips persist visually until Compile or Expand bakes them into the textarea — at which point the tray clears.

Chips don't persist across shot switches in v1; they live only in panel state. Storyboard-level prompt injection (apply a style sentence to every shot in a sequence) is planned as part of the storyboard system.

Disabling States

  • Compile is enabled when the prompt contains at least one #marker or there is at least one style chip.
  • Expand is enabled when the prompt has any non-empty text or at least one style chip.
  • Both buttons are disabled while a generation is running on the host tab.

Tag Marker Format

Markers are lowercased, slug-safe versions of the token name: spaces become _, and only letters/digits/underscore/hyphen are allowed in the body. A tag named Low Key becomes #low_key. LoRA markers come straight from each entry's trigger_word field, so however you spell the trigger in defaults_loras.json is the marker you type.

Marker collisions resolve in this order, first match wins: tags, then LoRA triggers. That means a user-authored tag of the same name overrides the auto-generated LoRA shorthand — useful when you want to swap a LoRA's default inject_text for your own phrasing without touching the LoRA config.

API Endpoints

  • GET /api/prompt/tokens?model=…&active_loras=… — unified token feed: tags + family-filtered LoRA triggers + caption phrases (only for active LoRAs). Returns { tokens, categories, model_family, active_lora_triggers }.
  • POST /api/prompt/compile — deterministic expansion. Body { text, model?, active_loras?, style_chips?, style_label? }, returns { compiled, expanded_markers, unknown_markers, missing_lora_triggers, style_sentence }.
  • POST /api/prompt/expand — LLM refinement. Body { text, model?, active_loras?, style_chips?, mode, intent? }, returns { positive, negative, notes?, model, mode }.

Describe & Tag

The Describe & Tag tool lives in the Utilities tab. It is the first slice of FUK's LLM integration: a vision-language model (Ollama-hosted) writes a cinematographer-style description of any image or short video clip, and lets you save fragments of that description as named prompt-shorthand tags for reuse later in the pipeline.

Requirements

  • Ollama running locally (default http://localhost:11434). FUK does not spawn or manage it — start ollama serve separately
  • The configured vision model pulled: ollama pull qwen3-vl:8b (≈6 GB on disk, ≈6–7 GB VRAM at runtime)
  • The health indicator at the top of the tool reports green when both conditions are met

Override the model with FUK_LLM_MODEL and the host with OLLAMA_HOST environment variables. Every request uses keep_alive: 0 so the LLM is evicted from VRAM immediately after each call, leaving room for the generation pipeline.

Describe an Image

  1. Click the drop zone to open a native file dialog and pick any image
  2. Click Describe Image — the image is downscaled to 1024px JPEG and sent to the model
  3. The result lands in an editable textarea so you can refine wording before tagging

Describe a Video

The same tool handles short video clips. Selecting a video file (.mp4 .mov .webm .mkv .avi .m4v) switches the preview to a playable video and the run button to Describe Video.

  • Six frames are sampled evenly across the clip at 512px each and sent as a single multi-frame request
  • A video-specific system prompt asks the model to treat the frames as one continuous shot and describe camera movement and how the action evolves — not the frames in isolation
  • The response includes the number of frames used, and the timeout extends to 120 s for the multi-frame call

Prompt Tags

Tags are reusable shorthand for chunks of prompt text. They live in fuk/ui/data/prompt_tags.json and persist across sessions. Future prompt-construction and storyboard features will expand tag names back into their full value when assembling prompts.

Saving a Tag from a Description

  1. Highlight any span of text in the description textarea (for example, the sentence describing the subject)
  2. Click Tag selection — the highlighted text drops into the tag form's Value field
  3. Give it a short name (e.g. fat man), pick a category, click Save Tag

The Tag all button uses the full description as the value instead. Tags can also be created manually by typing name and value directly.

Tag Fields

  • Name — 1–64 characters, letters/digits/space/underscore/hyphen. Stored lowercase, must be unique
  • Category — picked from the preset list (see below); optional, can be left as uncategorized
  • Value — up to 4000 characters of free text

Categories

Categories keep the tag list organized once it grows. The starter set:

  • composition — framing, shot type
  • subject — character/object descriptions
  • lighting — key, fill, mood, time of day
  • camera — angle, lens, movement
  • color — palette, grading
  • style — film stock, era, art direction
  • atmosphere — weather, ambient feel
  • setting — location, environment

The category list is user-editable at fuk/ui/data/prompt_tag_categories.json. The file is auto-seeded on first run; add, rename, or remove entries and reload the tab to refresh. Tags assigned to a category that you later remove will be flagged on save until you reassign them.

Managing Tags

The tag list at the bottom of the panel supports:

  • Category filter — dropdown with live counts per category and an uncategorized bucket
  • Text filter — matches against name, category, and value
  • Edit — loads a tag into the form for in-place changes
  • Delete — trash-can icon with a confirmation prompt

API Endpoints

Exposed under /api for use by future features and external tooling:

  • GET /api/llm/health — Ollama reachability and whether the configured model is pulled
  • POST /api/llm/describe — body { image_path }, dispatches image or video by extension, returns { description, kind, frames_used? }
  • GET /api/tags — list all tags, sorted by category then name
  • POST /api/tags — create { name, value, category? }
  • PUT /api/tags/{id} — update
  • DELETE /api/tags/{id} — remove
  • GET /api/tags/categories — the current category list

Generation History

The Generation History is a collapsible side panel available from any tab. It shows all previous generation outputs for the current session and provides drag-and-drop access to any result — the primary way to chain pipeline steps without manual file picking.

Panel Contents

  • Thumbnail grid of all generation results from the current session, most recent first
  • Each thumbnail shows the output type (image, video, preprocessed map, etc.) and basic metadata on hover: seed, model, timestamp
  • Video thumbnails play on hover at the global playback FPS

Drag and Drop

Any history item can be dragged onto:

  • A tab's preview area — reloads all metadata from that result into the active tab's parameter cards, including model, prompt, seed, LoRA settings, and dimensions
  • A Media Input component in any tab — inserts the file path directly as the input, without loading metadata

This is the standard workflow for passing results between pipeline steps — generate in Image, drag to Layers for AOV extraction, drag from Layers to Export for EXR assembly.

Playback Speed

The global playback speed control (6–48 FPS) in the app header affects video thumbnail playback in the history panel. Set it to match your generation frame rate for accurate motion preview.

Automatic per-VRAM-tier configuration. Three profiles — 24GB, 16GB, 12GB — each set specific combinations of precision flags and block swapping. FUK selects the appropriate profile based on the detected GPU and the active VRAM preset.

Metadata & Repeatability

Every generation in FUK captures a complete record of the parameters used. This metadata is written to a meta.json sidecar file alongside the output and is accessible by dropping the output onto any tab's preview area. A generation is fully reproducible from its sidecar alone, on any machine with the same model files.

Captured Metadata

FieldDescription
SeedExact seed used — fixed seeds with identical parameters produce identical results
ModelCheckpoint filename and SHA256 hash — hash confirms you're using the same file, not just the same name
LoRA(s)LoRA filenames, hashes, and per-LoRA blend weights
PromptFull positive and negative prompts, exactly as submitted
ParametersSteps, CFG scale, sampler, scheduler, resolution
ControlNetAny ControlNet models used, their preprocessor, weight, guidance range, and input image hash
EliGen ZonesZone mask paths, per-zone prompts, and weights if regional conditioning was used
PipelineWhich generation pipeline was used (base, controlnet, eligen, img2img, i2v, fun_control)
FUK VersionApplication version string
TimestampISO 8601 date and time of generation
Generation TimeWall-clock time for the generation in seconds

Metadata Format

Sidecar files are standard JSON, saved as meta.json in the version directory. They are human-readable and can be parsed by any standard JSON library. No database or index is required — the History panel reads sidecar files directly from disk.

Portability

Because metadata is stored as sidecar files alongside outputs rather than in a central database, output directories are fully portable. Copy a version directory to another machine and FUK will recognize it on first launch. If a model file referenced in the sidecar is missing or has a different hash, FUK will warn before re-queuing.
